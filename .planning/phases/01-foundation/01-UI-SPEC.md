# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Foundation.
> Generated by pivota_spec-ui-researcher, verified by pivota_spec-ui-checker.
>
> **Phase 1 UI Scope:** Login page, basic authenticated layout shell (navigation sidebar scaffold),
> and a minimal user role assignment screen (Admin only). The majority of this phase is backend
> (Docker, migrations, auth API, data seed). The UI delivered here is the minimal surface that
> lets a developer log in and confirm auth is working.

| Token | Value | Usage |

| sm | 8px | Compact element spacing; field label-to-input gap |

## Screen Specifications

### Screen 1: Login Page

**Source:** UX-Mockup-EMS.md — Screen-00

**Layout:**
- Full-viewport centered layout: white background (`hsl(0 0% 100%)`)
- Login card: 400px wide, white background, border (`hsl(214 32% 91%)`), shadow-md, border-radius 8px
- Card vertical padding: 48px (`2xl`); horizontal padding: 32px (`xl`)
- Logo/wordmark above card: "EMS" wordmark at 28px/700 + "Engagement Management System" at 14px/400 in muted text
- Footer copyright below card: 12px/400 muted text

**Field layout (inside card):**
- Card heading "Sign in to your account": 20px/600, centered, margin-bottom 24px
- Field label → input gap: 8px (`sm`)
- Input → next field gap: 16px (`md`)
- Last input → Sign In button gap: 24px (`lg`)
- Input height: 40px; border-radius: 6px; padding: 12px 16px

**Interaction rules (from UX Mockup):**
- `email` field auto-focuses on page load
- Enter on email field → moves focus to password field
- Enter on password field → submits form
- Show/hide password toggle: `lucide-react` Eye (hidden) / EyeOff (shown) icons, 16px, right-padding of input
- "Sign In" button disabled during loading state; shows spinner + "Signing in..." text
- No partial validation while typing — validation only triggers on submit attempt

**Error message design:**
- Error banner appears BELOW the Sign In button, INSIDE the card
- Role: `role="alert"` for screen reader announcement
- Generic message (no field disclosure): never reveal which field is wrong
- Missing field errors appear inline below each field (not in the banner)

## Accessibility Contract

*Source: PRD-EMS.md Section 7 — "UI must follow common accessibility practices for form labels,
keyboard navigation, and readable contrast."*

| Requirement | Specification |
|-------------|---------------|
| Color contrast | Minimum 4.5:1 for body text; 3:1 for large text (≥18px bold or ≥24px regular) |
| Focus indicators | 2px accent ring, 2px offset on all interactive elements; never `outline: none` without replacement |
| Form labels | All inputs have explicit `<label for>` association; no placeholder-as-label |
| Error messages | `role="alert"` on auth error banner; `aria-describedby` linking field to its error message |
| Button loading state | `aria-disabled="true"` + `aria-label="Signing in, please wait"` on loading Sign In button |
| Keyboard navigation | Tab order: Email → Password → Show/Hide Toggle → Sign In → Error; logical DOM order matches visual order |
| Screen reader | Nav landmarks: `<nav>`, `<main>`, `<header>` used correctly; active nav item has `aria-current="page"` |
| Password toggle | `aria-label="Show password"` / `aria-label="Hide password"` on toggle button |
