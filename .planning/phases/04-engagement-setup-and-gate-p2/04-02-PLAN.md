From TechArch F5 API spec:
```

PUT    /api/engagements/:id/milestones    // Upsert all 4 milestones at once
// Body: [{ milestone_type: 'planning_approval'|'evidence_readiness'|'draft_readiness'|'final_readiness', target_date: ISO8601 }]
// Computes status: not_started|on_track|at_risk|complete based on target_date vs today
```

From FRD validation rules:
- Team member removal: cannot remove the sole QA Reviewer if P2 has not yet been approved (HTTP 409, error code `TEAM_ROLE_REQUIRED`)
- Also cannot remove last EM from team (HTTP 409, code `TEAM_ROLE_REQUIRED`)
- Milestone status computation:
  - `complete` if `completed_at` is set
  - `not_started` if `target_date` is null
  - `on_track` if `target_date >= today + 7 days`
  - `at_risk` if `target_date` is in `[today, today + 6 days]`
  - `overdue` if `target_date < today`

**Types:**
```typescript
export type MilestoneType = 'planning_approval' | 'evidence_readiness' | 'draft_readiness' | 'final_readiness';
export type MilestoneStatus = 'not_started' | 'on_track' | 'at_risk' | 'overdue' | 'complete';
export type TeamRole = 'AL' | 'EM' | 'AN' | 'QA' | 'IR' | 'PC' | 'RO';

export interface TeamAssignment {
  id: string;
  engagement_id: string;
  user_id: string;
  role: TeamRole;
  assigned_at: string;
  user: { display_name: string; email: string };
}

export interface MilestoneRecord {
  id: string | null;
  engagement_id: string;
  milestone_type: MilestoneType;
  target_date: string | null;
  completed_at: string | null;
  status: MilestoneStatus;
}

export interface PrerequisiteItem {
  key: string;
  label: string;
  pass: boolean;
  detail?: string;
}
```

**Milestone status helper:**
```typescript
function computeMilestoneStatus(target_date: string | null, completed_at: string | null): MilestoneStatus {
  if (completed_at) return 'complete';
  if (!target_date) return 'not_started';
  const today = new Date();
  today.setHours(0, 0, 0, 0);
  const target = new Date(target_date);
  target.setHours(0, 0, 0, 0);
  const diffDays = Math.floor((target.getTime() - today.getTime()) / (1000 * 60 * 60 * 24));
  if (diffDays >= 7) return 'on_track';
  if (diffDays >= 0) return 'at_risk';
  return 'overdue';
}
```

**`listTeam(engagementId)`:**
- Query `team_assignments` JOIN `users` on `user_id = users.id`
- Select `team_assignments.*`, `users.display_name`, `users.email`
- Return `{ assignments: TeamAssignment[] }`

**`addTeamMember(engagementId, { user_id, role })`:**
- Validate `role` is one of the 7 allowed values (AL, EM, AN, QA, IR, PC, RO)
- Check for duplicate: query `team_assignments WHERE engagement_id = :id AND user_id = :user_id AND role = :role`
- If duplicate: throw `{ status: 409, message: 'This user already holds this role on the engagement.', code: 'DUPLICATE_ASSIGNMENT' }`
- Insert into `team_assignments`; return new `TeamAssignment` with user details

**`removeTeamMember(engagementId, assignmentId)`:**
- Find the assignment by id + engagement_id (404 if not found)
- Get role from the assignment
- If role is `QA`: check if P2 has been approved (query `gate_decisions WHERE engagement_id = :id AND gate_name = 'P2' AND decision = 'approved'`). If P2 NOT yet approved AND there is only 1 QA on the team: throw `{ status: 409, message: 'Cannot remove the QA Reviewer before Gate P2 has been approved.', code: 'TEAM_ROLE_REQUIRED' }`
- If role is `EM`: count remaining EM assignments (not counting this one). If count = 0: throw `{ status: 409, message: 'Cannot remove the last Engagement Manager from the team.', code: 'TEAM_ROLE_REQUIRED' }`
- Delete the assignment

**`listMilestones(engagementId)`:**
- Query `milestones WHERE engagement_id = :id`
- Return all 4 milestone types. If a milestone type has no row yet, return a synthetic object `{ id: null, engagement_id, milestone_type, target_date: null, completed_at: null, status: 'not_started' }`
- Always return exactly 4 items for the 4 milestone types: `planning_approval`, `evidence_readiness`, `draft_readiness`, `final_readiness`
- Compute `status` for each using `computeMilestoneStatus`

**`upsertMilestones(engagementId, dates: Array<{ milestone_type, target_date }>)`:**
- Validate: all milestone_types must be valid; target_dates must be parseable ISO8601 dates
- Date order validation: P2 target_date ≤ P3 target_date ≤ Draft target_date ≤ Final target_date (if all provided). If out of order: throw `{ status: 422, message: 'Milestone dates must be in chronological order.' }`
- For each provided milestone type: INSERT INTO milestones ... ON CONFLICT (engagement_id, milestone_type) DO UPDATE SET target_date = EXCLUDED.target_date, updated_at = NOW()
  (use `db.raw` or knex `onConflict` for upsert)
- Return all 4 milestones (same as listMilestones)

**`checkP2Prerequisites(engagementId)`:**
Check 7 prerequisites per FRD F7:
1. `has_objectives`: count objectives in `planning_record_objectives` (via `planning_records`) — pass if ≥1
2. `owner_assigned`: check `engagements.owner_id IS NOT NULL`
3. `has_em_on_team`: count `team_assignments WHERE role = 'EM'` — pass if ≥1
4. `has_qa_on_team`: count `team_assignments WHERE role = 'QA'` — pass if ≥1
5. `all_milestones_set`: count milestones with non-null target_date — pass if = 4
6. `risk_notes_present`: check `planning_records.risk_notes IS NOT NULL AND != ''`
7. `data_reliability_notes_present`: check `planning_records.data_reliability_notes IS NOT NULL AND != ''`

NOTE on independence_status: Per FRD "all team members have independence_status set" is a P2 prerequisite, but the `independence_affirmations` table is managed by the planning service (Plan 04-03). Include it as item 8 in the check — query `independence_affirmations WHERE engagement_id = :id` and compare count to team member count. If independence_affirmations table does not exist yet (Plan 04-03 not run), default this item to `pass: false` with detail "No independence affirmations recorded."
