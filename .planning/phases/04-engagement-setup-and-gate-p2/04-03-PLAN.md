**Types:**
```typescript
export type PlanningStatus = 'draft' | 'ready_for_review' | 'approved' | 'returned';
export type IndependenceStatus = 'affirmed' | 'pending' | 'exception_noted';

export interface PlanningRecord {
  id: string;
  engagement_id: string;
  status: PlanningStatus;
  design_approach: string | null;
  schedule_notes: string | null;
  risk_notes: string | null;
  data_reliability_notes: string | null;
  created_by: string;
  created_at: string;
  updated_at: string;
}

export interface Objective {
  id: string;
  planning_record_id: string;
  objective_text: string;
  information_need: string | null;
  sort_order: number;
  created_at: string;
}

export interface IndependenceAffirmation {
  id: string;
  engagement_id: string;
  user_id: string;
  status: IndependenceStatus;
  created_at: string;
  updated_at: string;
}
```

**`getPlanningRecord(engagementId)`:**
- Query `planning_records WHERE engagement_id = :id` (first row)
- If no record: return `{ planning_record: null, objectives: [], independence_affirmations: [] }`
- Query objectives: `SELECT * FROM planning_record_objectives WHERE planning_record_id = :pr_id ORDER BY sort_order ASC`
- Query affirmations: `SELECT * FROM independence_affirmations WHERE engagement_id = :id`
- Return `{ planning_record: PlanningRecord, objectives: Objective[], independence_affirmations: IndependenceAffirmation[] }`

**`upsertPlanningRecord(engagementId, data, actorId)`:**
- `data`: `{ design_approach?, schedule_notes?, risk_notes?, data_reliability_notes? }`
- If no existing record: INSERT with `status = 'draft'`, `created_by = actorId`
- If existing record and status is `approved`: do NOT overwrite — this function is only for draft edits. If user is editing after revision unlock (plan 04-03 handles unlock), status will still be `approved`; allow update but do not change status.
  - Actually: allow update regardless of status (the lock enforcement is UI-side; backend allows editing). This matches the FRD — the revision flow "unlocks" the record conceptually but the backend allows editing to support the revision.
- ON CONFLICT (engagement_id): UPDATE the fields that are non-null in data (use `COALESCE` or conditional update)
- Return updated `PlanningRecord`

**`submitPlanningRecord(engagementId)`:**
- Get existing planning record
- If none: throw `{ status: 404, message: 'No planning record found.' }`
- If status is not `draft` and not `returned`: throw `{ status: 409, message: 'Planning record can only be submitted from Draft or Returned status.' }`
- Update `status = 'ready_for_review'`
- Return updated `PlanningRecord`

**`addObjective(engagementId, { objective_text, information_need })`:**
- Validate objective_text is non-empty
- Get or create planning record (upsert with empty data)
- Compute next `sort_order`: MAX(sort_order) + 1 for this planning_record_id (default 1 if none)
- INSERT into `planning_record_objectives`
- Return new `Objective`

**`updateObjective(engagementId, objectiveId, { objective_text?, information_need?, sort_order? })`:**
- Verify objective exists and belongs to an engagement's planning record
- If objective_text provided: validate non-empty
- UPDATE the fields
- Return updated `Objective`

**`deleteObjective(engagementId, objectiveId)`:**
- Verify objective belongs to this engagement's planning record
- Check if objective has any linked evidence items: query `evidence_objective_links WHERE objective_id = :id`
- If linked evidence exists: throw `{ status: 409, message: 'Cannot delete this objective — it has linked evidence items. Unlink evidence first.' }`
- DELETE the objective

**`setIndependenceStatus(engagementId, affirmations: Array<{ user_id, status }>)`:**
- Validate each status is one of `'affirmed'|'pending'|'exception_noted'`
- For each entry: INSERT INTO `independence_affirmations` ... ON CONFLICT (engagement_id, user_id) DO UPDATE SET status = EXCLUDED.status, updated_at = NOW()
- Return all affirmations for this engagement

**`recordP2Decision(engagementId, { decision, comment }, actorId)`:**
- Validate: decision must be `'approved'` or `'returned'`
- Validate: comment required and ≥10 chars (throw `{ status: 422, message: 'Decision comment must be at least 10 characters.' }`)
- Re-run prerequisites: import `checkP2Prerequisites` from `'./team.service'`. Call it for this engagement.
  - If `all_pass = false`: throw `{ status: 422, message: 'P2 prerequisites not met.', failed_prerequisites: prerequisites.filter(p => !p.pass) }`
- Planning record must be in `ready_for_review` status: if not, throw `{ status: 409, message: 'Planning record is not submitted for review.' }`
- Execute in a **`db.transaction()`**:
  1. INSERT into `gate_decisions`: `{ engagement_id, gate_name: 'P2', decision, rationale: comment, decided_by: actorId, decided_at: NOW() }`
  2. INSERT into `audit_events`: `{ engagement_id, action_code: decision === 'approved' ? 'GATE_P2_APPROVED' : 'GATE_P2_RETURNED', actor_id: actorId, metadata: JSON.stringify({ comment }) }`
  3. If `approved`:
     - UPDATE `planning_records SET status = 'approved', updated_at = NOW()` WHERE engagement_id
     - UPDATE `engagements SET phase = 'evidence', updated_at = NOW()` WHERE id = engagementId
  4. If `returned`:
     - UPDATE `planning_records SET status = 'returned', updated_at = NOW()` WHERE engagement_id
- Return `{ gate_decision, planning_record }`

**`requestRevision(engagementId, { revision_note }, actorId)`:**
- Validate revision_note is ≥10 chars
- Get planning record — must exist and be `status = 'approved'`; throw 409 if not
- INSERT into `audit_events`: `{ engagement_id, action_code: 'PLANNING_REVISION_REQUESTED', actor_id: actorId, metadata: JSON.stringify({ revision_note }) }`
- NOTE: In the DB schema (Phase 1 migration 002), the planning_revisions table stores revision notes. INSERT into `planning_revisions` if that table exists. Check migration 002 for the exact table name and columns; if it's `planning_revisions`, insert `{ planning_record_id, requested_by: actorId, revision_note, created_at: NOW() }`.
- Return updated `PlanningRecord` (status unchanged at `approved` — the unlock is conceptual)
