  <action>
Create `backend/src/services/statements.service.ts` implementing all draft statement business logic.

Follow the same service pattern as `backend/src/services/evidence.service.ts`.

**Exact API contracts from TechArch:**

```
GET    /api/engagements/:id/statements         // List statements (query: ref_status, assigned_to)
POST   /api/engagements/:id/statements         // Create statement
// Body: { statement_text: string, evidence_ids: string[] }
// Validates: ≥1 evidence_id required → 422 if 0

PATCH  /api/engagements/:id/statements/:statement_id  // Update statement
// Body: { statement_text?, evidence_ids?, ref_status?, discrepancy_notes?, discrepancy_type?, assigned_to?, assigned_back_to? }

DELETE /api/engagements/:id/statements/:statement_id  // Delete statement
```

**draft_statements table columns** (from 002 migration — use exact column names):
`id` (uuid), `engagement_id`, `statement_text`, `ref_status` (default: 'not_started'), `sequence_number` (auto-increment per engagement), `assigned_to` (IR user_id), `assigned_back_to` (AN user_id), `discrepancy_notes`, `discrepancy_type`, `created_by`, `created_at`, `updated_at`

**statement_evidence_links table** (junction):
`id` (uuid), `statement_id`, `evidence_id`, `created_at`

**listStatements:**
- Join with statement_evidence_links to compute evidence_count and evidence_ids[]
- Join with users table for assigned_ir_name (assigned_to → user displayname) and assigned_an_name (assigned_back_to → user displayname)
- Default sort order (from UI-SPEC): not_started FIRST, then in_review, then failed, then passed, then waived
  - Use: `ORDER BY CASE ref_status WHEN 'not_started' THEN 0 WHEN 'in_review' THEN 1 WHEN 'failed' THEN 2 WHEN 'passed' THEN 3 WHEN 'waived' THEN 4 END`
- Filter by ref_status (optional query param, comma-separated)
- Filter by assigned_to (optional, for IR queue filtering)

**createStatement:**
- Validate evidence_ids.length >= 1; throw 422 "At least one evidence item must be linked to a statement."
- Validate each evidence_id exists in evidence_items table for this engagement
- Insert into draft_statements with sequence_number = (SELECT COUNT(*) FROM draft_statements WHERE engagement_id = :id) + 1
- Insert into statement_evidence_links for each evidence_id
- Return statement with evidence_ids and evidence_count

**updateStatement:**
- evidence_ids: if provided, replace all links (delete old statement_evidence_links, insert new)
- ref_status = 'waived': requires `userRoles.includes('EM') || userRoles.includes('AD')` — otherwise 403; requires discrepancy_notes length ≥ 10 chars — otherwise 422
- ref_status = 'failed': requires discrepancy_notes non-empty — 422 if empty "Discrepancy notes are required when reference status is Failed."
- assigned_back_to (discrepancy assignment): if provided and ref_status = 'failed', create AuditEvent:
  ```
  db('audit_events').insert({
    id: uuid(),
    engagement_id: engagementId,
    actor_id: userId,
    action_code: 'STATEMENT_DISCREPANCY_ASSIGNED',
    target_type: 'statement',
    target_id: statementId,
    metadata: JSON.stringify({ assigned_to: assigned_back_to, statement_id: statementId }),
    created_at: new Date()
  })
  ```
- Return updated statement with fresh evidence_ids join

**deleteStatement:**
- If ref_status IN ('passed', 'waived') → 409 "Cannot delete — this statement is linked to a completed reference check. Change status first."
- Otherwise: delete from statement_evidence_links (FK cascade or manual), delete from draft_statements

// GET /api/engagements/:id/statements — all authenticated roles
statementsRouter.get('/', listStatementsHandler)
