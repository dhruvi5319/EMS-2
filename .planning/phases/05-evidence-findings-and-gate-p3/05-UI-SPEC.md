# Phase 5 — UI Design Contract

> Visual and interaction contract for Phase 5: Evidence, Findings, and Gate P3.
> Generated by pivota_spec-ui-researcher, verified by pivota_spec-ui-checker.
>
> **Phase 5 UI Scope:** Evidence Registry page (F8), Add/Edit Evidence form with multi-file upload,
> Evidence Detail page, evidence-to-objective linking panel (F9), Objective Coverage / Gap view (F9),
> CSV export, Findings list and Add/Edit Finding form (F10), Objective Sufficiency panel (F10 — QA),
> Gate P3 Review panel (F10 — QA), and restricted evidence access indicators.
>
> **Critical constraint:** This spec EXTENDS Phases 1–4 — all spacing, typography, color, layout
> constants, and design system tokens are LOCKED and must not be re-declared with different values.
> Phase 5 introduces new shadcn components, new screen layouts, new interaction patterns, and new
> semantic colors for P3 sufficiency status without contradicting any prior phase.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline label padding, badge internal padding |
| sm | 8px | Compact element spacing; field label-to-input gap; char counter gap |
| md | 16px | Default element spacing; card internal padding; form field gap |
| lg | 24px | Section padding; card-to-card gap; evidence form section separator spacing |
| xl | 32px | Layout gaps; form container vertical padding |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing; top bar height |

Phase 5 exceptions (new additions — no Phase 1/2/3/4 contradictions):
- Evidence file upload drop zone min-height: 120px (consistent with Phase 3 `IntakeFileUpload`)

- Evidence file upload progress bar height: 4px (matches Phase 3 FileProgressBar)

- Objective coverage progress bar height: 8px (thin horizontal bar showing covered/total)
- Add Evidence form max-width: 640px (slide-in panel width at desktop)
- Add Finding form max-width: 520px (modal max-width)
- P3 decision panel left border: 4px solid (accent — matches Phase 3 A1 panel and Phase 4 P2 decision panel)
- Evidence detail two-column field grid: 2-column (label left, value right) at ≥768px (matches Phase 3 request detail)
- Gap view objective row height: 48px (touch-target compliant)
- Touch target minimum: 44×44px on all interactive controls (inherited)

- Evidence ID (e.g., `E-001`): monospace 14px/400 — `ui-monospace, 'Cascadia Code', 'Source Code Pro', monospace`
- Finding ID (e.g., `FD-001`): monospace 14px/400 — same monospace stack
- File size in file list: 12px/400 muted color
- Evidence type badge text: 12px/400 label role
- Sensitivity badge text: 12px/600 semibold (weight elevated for RESTRICTED label — safety signal)
- P3 sufficiency chip text: 12px/400 label role (consistent with milestone status chip from Phase 4)
- Coverage progress bar label: 12px/400 muted — "X covered · Y gap"
- Objective text in gap view (truncated): 14px/400, max 2 lines (overflow ellipsis at 100 chars)
- Finding text in list (truncated): 14px/400, max 3 lines (overflow ellipsis, expandable)
- "RESTRICTED" badge text: 12px/600, uppercase, letter-spacing: 0.05em (matches Phase 3 FormSectionHeader uppercase treatment)
- Form section labels in evidence/finding forms: 12px/400 uppercase tracking-wide (inherits Phase 3 FormSectionHeader pattern)

**Evidence table columns:**

| Column | Content | Sortable | Notes |
|--------|---------|----------|-------|
| ID | `E-NNN` format (monospace 14px/400) | Yes | Short ID |
| Type | `EvidenceTypeBadge` (12px pill) | Yes | Document/Dataset/Interview Note/Meeting Note/Other |
| Source | Truncated at 40 chars (14px/400) | Yes | Full source in tooltip on hover |
| Date Received | MM/DD/YYYY (14px/400) | Yes | — |
| Sensitivity | `SensitivityBadge` | Yes | STANDARD (gray) or RESTRICTED (red, 12px/600 uppercase) |
| Obj (Objectives Linked) | Count integer (14px/400); 🔴 if 0 | No | Tooltip: "N objectives linked" |
| File | Count integer (14px/400) | No | — |
| Actions | `[View →]` accent link | No | Routes to Evidence Detail |

**Default sort:** `date_received` descending (most recently received first)

**Table row click:** Navigates to Evidence Detail page

**Restricted row visibility for AL/RO:** Row is completely absent from the DOM (not hidden with CSS). The evidence count shown in the header and coverage bar excludes restricted items for AL/RO.

**Objective coverage bar:**
- 8px tall horizontal bar
- Filled (blue-100, accent border) portion = objectives with ≥1 linked evidence item
- Gap (red-100) portion = objectives with 0 linked evidence
- Label: "X covered · Y gap" (12px/400 muted)
- [Show Gaps ▼] toggle link (14px/400 accent) — expands gap view below

**[+ Add Evidence] button:**
- Style: primary (accent fill, white text), 40px height
- Access: AN and AD only; EM sees button but is prevented at save (or server returns 403)
- Opens: `AddEvidencePanel` (slide-in Sheet from right edge, 640px)

**[Export CSV ↓] button:**
- Style: secondary/outline, 36px height
- Access: AL, EM, AN, QA, PC, RO, AD (IR excluded — per US-9.4)
- Restricted items excluded from export for AL and RO
- On click: immediate download trigger → toast "Evidence registry exported."
- Loading: spinner inside button

**Table loading state:** 5 `SkeletonRow` components (48px height each)

**Table empty state (no evidence):**
- Icon: `Archive` (lucide-react, 32px, muted)
- Heading: "No evidence items yet."
- Body: "Add your first evidence item to begin building the evidence record for this engagement."
- CTA: `[+ Add Evidence]` (AN/AD only — accent fill)

**Table empty state (filter active, no results):**
- Icon: `SearchX` (lucide-react, 32px, muted)
- Heading: "No evidence matches your filters."
- Body: "Try different filters or clear them to see all evidence."
- CTA: `[Clear filters]` (ghost)

## Visuals

### Visual Hierarchy

Each major surface has one primary visual anchor:

- **Primary visual anchor (Evidence Registry):** the `[+ Add Evidence]` accent-filled button and the `ObjectiveCoverageBar` — coverage at a glance; gaps immediately signal action needed.
- **Primary visual anchor (Gap View):** the red dashed-border `GapObjectiveCard` components — red draws the eye to uncovered objectives with explicit P3 blocker labels.
- **Primary visual anchor (Findings Tab):** the `ObjectiveSufficiencySummary` bar at the top — the P3 sufficiency chips and "BLOCKED/READY" status line orient QA and analysts to the gate state immediately.
- **Primary visual anchor (P3 Review Panel):** the P3 prerequisites checklist — QA sees immediately whether approval is possible. The `[✓ Approve P3]` accent button at the bottom is the terminal action.
- **Primary visual anchor (Evidence Detail):** the `SensitivityBadge` in the page header — RESTRICTED in red immediately signals access restrictions.

All other interactive elements use secondary (outline), ghost, or muted styles.
