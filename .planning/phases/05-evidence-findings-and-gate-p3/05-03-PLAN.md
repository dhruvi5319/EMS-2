# Plan 05-03: Findings + Gate P3 API

**Phase:** 5 — Evidence, Findings, and Gate P3  
**Status:** not started  
**Features:** F10  

## Goal

Implement findings CRUD with evidence linking, P3 prerequisites check, and the Gate P3 approval decision.

## Tasks

### Task 1 — Findings API
- Create `src/modules/findings/findings.repository.ts`, `findings.service.ts`, `findings.controller.ts`:
  - `GET /api/engagements/:id/findings` — list findings with linked evidence
  - `POST /api/engagements/:id/findings` — create finding (AN, EM, AD); must link ≥1 evidence item
  - `PATCH /api/engagements/:id/findings/:findingId` — update finding text/status
  - `DELETE /api/engagements/:id/findings/:findingId` — soft delete (blocked if referenced by statements)
  - `POST /api/engagements/:id/findings/:findingId/evidence` — link evidence items (array of evidence_item_ids)
  - `DELETE /api/engagements/:id/findings/:findingId/evidence/:evidenceId` — unlink

### Task 2 — P3 prerequisites and gate decision
- Create `src/modules/gates/p3.prerequisites.ts`:
  - `checkP3Prerequisites(engagementId)` returns `{ met: boolean, missing: string[] }`:
    1. P2 gate has status=approved
    2. All objectives have ≥1 linked evidence item
    3. No objective has sufficiency_status = evidence_needed
    4. At least one finding exists (optional — verify against FRD)
- Add to gates routes (on engagementsRouter to avoid path conflicts with /:finding_id):
  - `GET /api/engagements/:id/gate/p3/prerequisites`
  - `POST /api/engagements/:id/gate/p3/approve` (QA, AD):
    - Server-side re-validates all P3 prerequisites
    - Creates GateDecision (gate_type=P3, status=approved)
    - Sets engagement.phase = draft
    - Creates AuditEvent (GATE_P3_APPROVED)
    - Idempotency guard: returns 409 if P3 already approved
