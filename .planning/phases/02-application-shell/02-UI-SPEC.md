# Phase 2 — UI Design Contract

> Visual and interaction contract for Phase 2: Application Shell.
> Generated by pivota_spec-ui-researcher, verified by pivota_spec-ui-checker.
>
> **Phase 2 UI Scope:** Full navigation sidebar with role-based filtering, global search bar
> and results overlay, user management page (Admin only), role assignment UI, per-engagement
> audit trail view, empty states for all major list views, and a 403 forbidden page.
>
> **Critical constraint:** This spec EXTENDS Phase 1 — all spacing, typography, color, and
> layout constants are inherited. Nothing in Phase 1 UI-SPEC is contradicted here. Phase 2
> adds the full RBAC nav layer, shadcn initialization, and new component contracts.

## Spacing Scale

Inherited from Phase 1 — no changes. Declared here for executor reference:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline label padding, badge internal padding |
| sm | 8px | Compact element spacing; field label-to-input gap |
| md | 16px | Default element spacing; card internal padding; form field gap |
| lg | 24px | Section padding; card-to-card gap; sidebar section breaks |
| xl | 32px | Layout gaps; form container vertical padding |
| 2xl | 48px | Major section breaks; login card vertical padding |
| 3xl | 64px | Page-level spacing; top bar height |

Exceptions (inherited from Phase 1, no additions):
- Sidebar width expanded: 220px (layout constant, not token)
- Sidebar width collapsed: 56px (icon-only; tablet breakpoint)
- Touch target minimum: 44×44px on all interactive controls (WCAG 2.5.5)
- Review Queue sidebar badge: 20px diameter pill (count badge)
- Search overlay max-width: 640px centered in content area
- Modal max-width: 600px (from UX Mockup responsive spec)

Inherited from Phase 1 — no changes. Declared here for executor reference:

**Notes (Phase 2 additions):**
- Audit trail action codes (e.g., `GATE_P4_APPROVED`): monospace at 12px/400 —
  `ui-monospace, 'Cascadia Code', 'Source Code Pro', monospace`

- Empty state heading: 20px/600 — use Heading role (`text-xl font-semibold`)
- Empty state body: 14px/400, muted color (`hsl(215 16% 47%)`)

## Screen Specifications

### Screen A: Full Navigation Sidebar (Role-Filtered)

**Source:** UX-Mockup-EMS.md — Layout Shell, Information Architecture

**Layout constants (inherited from Phase 1):**
- Sidebar width (expanded): 220px fixed left
- Sidebar width (collapsed): 56px icon-only (tablet ≤1024px)
- Background: `hsl(210 20% 97%)` (secondary color)
- Left border: 1px solid `hsl(214 32% 91%)`

**Nav sections (all 6 — filtered per role):**

| Section | Icon (lucide-react) | Shown to roles |
|---------|---------------------|----------------|
| Dashboard | `LayoutDashboard` | All roles |
| Requests | `FileText` | AL, EM, RO, AD |
| Engagements | `Briefcase` | All roles |
| Evidence | `Archive` | AN, EM, QA, IR, PC, AD |
| Review Queue | `ClipboardCheck` | AL, QA, IR, PC, AD |
| Reports | `BarChart2` | AL, EM, QA, PC, RO, AD |

> **Role filtering logic:** The `RoleFilteredNav` component receives `currentUser.roles[]` and
> renders only sections in the user's allowed set. A user with multiple roles sees the union of
> all allowed sections. AN does NOT see Requests. RO does NOT see Review Queue.

**Nav item anatomy (each item):**
- Height: 40px; left padding: 16px (`md`)
- Icon: 16px lucide-react, `gap-x-8px` from text label
- Text: 14px/400 (body)
- Active state: 4px left accent bar (`hsl(221 83% 53%)`), text in accent, background `hsl(210 40% 96%)`
- Hover state: background `hsl(210 40% 96%)`, cursor pointer
- Inactive icon color: `hsl(215 16% 47%)` (muted)
- Active icon color: `hsl(221 83% 53%)` (accent)
- `aria-current="page"` on active item

**Review Queue badge:**
- 20px diameter pill, background `hsl(221 83% 53%)` (accent), text white 12px/400
- Positioned right-aligned within the nav item row
- `aria-label="Review Queue, {N} items pending"` on the nav item
- Hidden when count is 0 (no empty badge shown)

**Bottom of sidebar:**
- Separator line: 1px `hsl(214 32% 91%)`
- Current user role badge(s): `RoleBadge` chips for each role code (AL, EM, etc.)
- User display name: 14px/400, muted color

**Sidebar collapse (tablet ≤1024px):**
- Width: 56px; shows icons only (no text labels)
- Hover/focus reveals section name as tooltip (12px, dark surface, 4px padding)
- `aria-label` on each icon-only nav item = section name

**Mobile (<768px):**
- Sidebar hidden; hamburger `≡` button in top bar triggers full-screen overlay
- Overlay: full viewport width, white background, closes on nav item tap or `✕`
- All 6 section names shown as full text; role badge and user name at bottom

### Screen B: Global Search Bar and Results Overlay

**Source:** UX-Mockup-EMS.md — Layout Shell, Screen-01 search pattern, US-0.4

**Top bar search input:**
- Position: horizontally centered in top bar
- Width: 320px at desktop; 100% minus padding at tablet/mobile
- Height: 36px (secondary button height from Phase 1 layout constants)
- Placeholder: "Search engagements, requests..." 14px muted
- Icon: `Search` (lucide-react, 16px) left inside input
- Border: 1px `hsl(214 32% 91%)`, radius 6px
- Focus: 2px accent ring, overlay opens
- Keyboard shortcut hint: `⌘K` / `Ctrl+K` in placeholder at desktop (right-aligned, 12px muted)

**Search trigger rules:**
- Min 2 characters before search fires
- Debounce: 300ms after last keystroke
- Fewer than 2 characters: show validation message "Type at least 2 characters to search."
- Empty input: overlay closes

**Results overlay:**
- Component: shadcn `Command` (or `Popover` over `Command`)
- Position: directly below search input, left-aligned
- Width: 640px max, full-width on mobile
- Background: `hsl(0 0% 100%)` white, border `hsl(214 32% 91%)`, shadow-lg, border-radius 8px
- Max height: 480px with scroll
- Backdrop: none (overlay only — no full-screen backdrop)
- Escape: closes overlay, returns focus to search input

**Results overlay states:**

| State | Content |
|-------|---------|
| Loading | Single skeleton row (shimmer, 40px height × 3) |
| Results | Up to 50 results; grouped if needed |
| No match | "No engagements found for "{query}"." + "Try different keywords." |
| Min chars not met | "Type at least 2 characters to search." (muted, 14px) |

**Each search result item:**
- Height: 48px (touch-target compliant)
- Left: `Briefcase` or `FileText` icon (16px muted) — Engagement vs Request
- Primary text: Title (14px/600 with match highlight) — max 60 chars truncated
- Secondary text: ID + Phase badge + Owner (12px/400 muted)
- Hover: `hsl(210 40% 96%)` background
- Enter or click: navigates to Engagement Shell or Request Detail

**Search result item layout:**
```
[icon]  Title of Engagement (match highlighted)        →
        ENG-2026-00001  [Planning]  Diana Okafor
```

**Results are scoped to engagements the user is authorized to view** (API enforces this — no UI
filtering needed; results returned by API are already role-scoped).

### Screen C: User Management Page (Admin Only)

**Source:** US-0.6, PRD Section 6.1, UX-Mockup-EMS.md AdminPage reference

**Route:** `/admin/users`

**Access control:** Only users with role `AD` can access. Direct URL access by non-AD returns 403 page.

**Layout:**
- Page header: "User Management" (h1, 20px/600)
- Breadcrumb: "Admin > User Management"
- Top-right action: `[+ Create User]` button (primary, accent fill)
- Main content: `UserTable` below header

**User table columns:**

| Column | Content | Sortable |
|--------|---------|----------|
| Name | `full_name` (14px/400) | Yes |
| Email | `email` (14px/400 muted) | Yes |
| Roles | `RoleBadge` chips for each role (12px, 4px radius pills) | No |
| Status | `UserStatusBadge`: "Active" (green) / "Deactivated" (gray) | Yes |
| Actions | [Edit Roles] [Deactivate] / [Reactivate] buttons | No |

**Table loading state:** 5 `SkeletonRow` components (shimmer, 48px height each)

**Table empty state:**
- "No users found. Create the first user to get started."
- `[+ Create User]` button below (only shown in empty state, Admin only)

**Create User dialog (modal):**
- Trigger: `[+ Create User]` button
- shadcn `Dialog`, max-width 520px
- Heading: "Create New User" (20px/600)
- Fields:
  - Full Name * (text input, 40px height)
  - Email Address * (email input, 40px height)
  - Password * (password input with show/hide toggle, 40px height)
  - Roles * (RoleCheckboxGroup — 8 checkboxes: AL, EM, AN, QA, IR, PC, RO, AD)
    — must select at least one; no selection shows "At least one role is required."
 - Actions: `[Discard]` (ghost) + `[Create User]` (primary, disabled until valid)
- Success: dialog closes + toast "User created successfully." (green, 4s)
- Error: inline error below field or dialog-level error banner

**Edit Roles dialog:**
- Trigger: `[Edit Roles]` in table row
- Same `RoleCheckboxGroup` as Create, pre-populated with current roles
- Heading: "Edit Roles — {user full_name}" (20px/600)
 - Actions: `[Discard Changes]` (ghost) + `[Save Roles]` (primary)
- Success: dialog closes + toast "Roles updated." (green, 4s)

**Deactivate user confirmation:**
- Trigger: `[Deactivate]` in table row
- shadcn `AlertDialog` (cannot dismiss by clicking outside)
- Title: "Deactivate {user full_name}?"
- Body: "They will no longer be able to log in. This can be reversed."
- Buttons: `[Cancel]` (ghost) + `[Confirm Deactivate]` (destructive, red)
- Success: row updates to "Deactivated" badge + toast "User deactivated." (green, 4s)

**Source:** UX-Mockup-EMS.md — Screen-14, US-0.5

 - `[Apply Filters]` primary-outline button; `[Clear]` ghost button

**Event card anatomy (each card):**
- Background: white with 1px border `hsl(214 32% 91%)`, border-radius 6px
- Padding: 16px (`md`) all sides

  ```
  Apr 30, 2026  14:22:07          Tom Andrade (PC)   [PC] badge
  GATE_P4_APPROVED                ← monospace 12px/400, muted bg pill
  ```

**Timeline pagination:**
- Show 20 events per page
- `[Load More]` button at bottom (secondary/outline): "Load More (showing 20 of 47)"
- `aria-busy="true"` on timeline container while loading

**Empty state:**
- "No audit events recorded yet." (muted 14px)
- No action button (audit is system-generated; user cannot add events)

**Filter empty state:**
- "No events match your filter." + `[Clear filters]` link (accent, 14px)

### Screen E: 403 Forbidden Page

**Source:** US-0.3 acceptance criteria: "direct URL access to a hidden section returns HTTP 403"

**Trigger:** User navigates directly to a route they don't have access to (e.g., `/admin/users`
for non-Admin, or `/engagements/{id}` for unauthorized user).

**Layout:**
- Full content area (sidebar + topbar remain visible)
- Centered content block: max-width 480px, vertically centered
- Icon: `ShieldX` (lucide-react, 48px, destructive color `hsl(0 72% 51%)`)
- Heading: "Access Denied" (20px/600)
- Body: "You don't have permission to view this page. Contact your administrator if you believe this is an error." (14px/400 muted)
- Back link: "← Go back to Dashboard" (accent color link, 14px)
- No retry button (permission issues require admin action)
